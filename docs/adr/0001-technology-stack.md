# ADR 0001 — Initial Technology Stack

Status: Accepted for the first local product phase
Date: 2026-07-12

## Context

JOLT must be local-first, single-user, Windows-friendly, auditable, and highly automated. Normal development must run in GitHub CI and a reproducible local environment. Live LinkedIn capture is supervised and local; fixture-based tests must cover the adapter contract without credentials.

## Decision

### Backend

- Python 3.12.
- FastAPI for the local HTTP API.
- Pydantic v2 for API schemas and validated configuration.
- SQLAlchemy 2 for persistence boundaries.
- SQLite for local transactional storage.
- Alembic for schema migrations.
- pytest and HTTPX/FastAPI TestClient for automated tests.
- Ruff for linting and formatting.
- Pyright for static type checking.
- uv for Python environment and dependency management.

### Frontend

- React with TypeScript.
- Vite for development and production builds.
- Vitest and React Testing Library for component tests.
- Playwright for end-to-end tests, screenshots, traces, and later supervised source-adapter investigation.

### Automation

- GitHub Actions for pull-request verification.
- Stable repository PowerShell runners for Windows.
- Synthetic and sanitized fixtures for normal tests.
- CI artifacts for logs, test reports, screenshots, and Playwright traces.

## Why this fits

- Python is suitable for parsing, normalization, evidence processing, exports, and browser automation.
- FastAPI provides a typed local API and straightforward pytest-based verification.
- SQLite keeps the product local while providing transactions, constraints, indexing, and migrations that JSON files cannot provide safely as the model grows.
- React provides an appropriate review-queue and application-workflow interface.
- Playwright supports both JOLT UI verification and controlled source-adapter testing.
- uv gives a fast, reproducible Python workflow while retaining standard `pyproject.toml` metadata.

## Constraints

- No cloud database or hosted backend in the first phase.
- No credential storage in the repository, database, logs, fixtures, or CI.
- No live LinkedIn access in routine CI.
- No Electron or desktop wrapper until the browser-based local application is stable.
- No Redux unless actual state complexity demonstrates the need.
- No LLM dependency in the canonical intake/evaluation pipeline. External AI review occurs through explicit analysis-pack export.

## Alternatives considered

### Python-only desktop UI

Rejected for now. It would simplify deployment but make the dense review, evidence, queue, and market-analysis interface harder to evolve and test than a browser UI.

### Node/TypeScript backend

Viable, but Python better matches the parsing, evidence, automation, export, and existing project knowledge needs. A single language across the whole stack is not valuable enough to override those advantages.

### JSON/JSONL persistence

Rejected as the canonical store. Updates, relationships, deduplication, historical evaluations, application events, and market analysis require transactions and structured queries.

### PostgreSQL

Deferred. It adds an unnecessary service and deployment burden for the initial local single-user product.

## Consequences

- Backend and frontend remain separate development processes.
- API contracts must be explicit and tested.
- Dependency lock files must be committed once generated by the automated scaffold workflow.
- Database migrations become mandatory after the first persisted schema is introduced.
- Live-source automation remains an adapter outside the domain core.
